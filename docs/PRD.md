## 4. 플랫폼 및 배포 전략

### 4.1 MVP 플랫폼

MVP는 **PWA 웹앱**으로 유저 테스트를 진행한다.

사용자는 모바일 브라우저에서 Qling에 접속하고, 필요 시 홈 화면에 추가하여 앱처럼 사용할 수 있다.

앱처럼 사용하기 안내 화면 또는 별도 설치 안내 영역은 MVP에서 제외한다.

PWA 설치, 공유, QR 안내는 추후 배포 및 온보딩 개선 단계에서 재검토한다.

### 4.2 추후 배포 계획

초기 유저 테스트 이후, Capacitor를 활용해 PWA를 래핑하고 iOS 및 Android 앱 배포를 검토한다.

## 5. 주요 사용자

Qling에서는 사용자가 고정적으로 “게시자” 또는 “답변자”로 나뉘지 않는다.

한 사용자는 상황에 따라 두 역할을 모두 수행할 수 있다.

- 고민을 작성할 때는 **고민 게시자**
- 다른 사람의 고민에 답변할 때는 **고민 답변자**

## 6. 핵심 가치 제안

### 6.1 공개 게시판이 없는 고민 공유

Qling에는 공개 게시판이 없다.
사용자가 작성한 고민은 전체 사용자에게 공개되지 않고, AI 매칭을 통해 선택된 답변자에게만 전달된다.

이를 통해 사용자는 다음 부담을 줄일 수 있다.

- 인터넷에 무분별하게 고민이 공개되는 부담
- 불특정 다수의 악성 댓글 가능성
- 공개 커뮤니티에서 평가받는 느낌

### 6.2 AI 맞춤 매칭

Qling은 고민 게시자의 고민 내용을 바탕으로 카테고리를 추론하고, 답변자 후보의 관심 분야, 좋아요 수, 성별을 활용해 답변자를 매칭한다.

매칭 우선순위는 다음과 같다.

1. 관심 분야
2. 좋아요 수
3. 성별

성별은 1, 2순위가 동률일 때 **같은 성별 우선 tie-breaker**로 사용한다.

나이는 MVP 매칭 우선순위에 사용하지 않는다. 나이는 사용자 프로필 데이터와 운영 분석용 데이터로 저장하되, 매칭 후보 정렬·필터링에는 반영하지 않는다.

### 6.3 AI 필터링

고민 게시, 답변, 좋아요/싫어요 코멘트에는 AI 필터링을 적용한다.

부적절한 내용은 전송되지 않으며, 사용자는 사유별 안내 문구를 받는다.

기본 차단 사유는 다음 6개로 시작한다.

1. 공격적/혐오/욕설
2. 성적 표현
3. 자해/자살 위험
4. 범죄/폭력 피해
5. 개인정보 포함
6. 스팸/홍보

고위험 고민은 게시를 차단하되, 일반 부적절 문구와 함께 도움 안내 문구를 제공한다.

## 7. MVP 필수 기능

## 7.1 온보딩

### 기능 설명

사용자는 첫 실행 시 닉네임, 성별, 나이, 관심 분야를 입력 또는 선택한다.

온보딩 순서는 다음과 같다.

1. 닉네임 입력
2. 닉네임 중복 확인
3. 성별/나이 입력
4. 관심 분야 선택
5. 예제 고민 생성
6. 답변하기 탭 진입

### 입력 항목

닉네임:

- 필수 입력값이다.
- 다른 사용자와 중복은 허용하지 않는다.
- 서버에서 최종 중복 검사를 보장한다.
- 닉네임 정규화 및 reservation document 등 구체적 구현 방식은 PRD가 아니라 구현 문서와 TODO에서 관리한다.
- 순위 탭에서는 닉네임을 공개한다.
- 고민, 답변, 채팅 맥락에서는 다른 사용자에게 보이지 않는다.
- 닉네임은 내가 쓴 답변 목록에서도 보이지 않는다.
- 닉네임은 본인의 마이페이지 프로필 summary와 순위 탭에서만 표시한다.
- 답변 목록, 고민 상세, 답변 상세, 예제 고민, 알림 문구, 채팅 화면에서는 닉네임을 노출하지 않는다.

성별:

- 남성
- 여성

성별은 필수다.
성별 정보는 매칭 품질 및 데이터 수집 목적을 위해 사용한다.

나이:

- 필수 입력값이다.
- 숫자로 입력받는다.
- 허용 범위는 만 14세 이상 99세 이하로 한다.
- 나이는 MVP 매칭 로직에는 사용하지 않는다.
- 나이는 사용자 프로필 데이터와 운영 분석용 데이터로 저장한다.
- 나이는 다른 사용자에게 절대 보이지 않는다.

관심 분야:

- 취업
- 진로
- 학업
- 시험
- 소득
- 주거
- 연애
- 결혼
- 부모
- 자녀
- 우울
- 불안
- 외로움
- 직장
- 워라밸
- 외모
- 자존감
- 건강
- 노후
- 미래
- 잡담

관심 분야는 최소 1개 이상 선택해야 한다.
복수 선택 가능하며, 이후 마이페이지에서 무제한 수정할 수 있다.

관심 분야 카테고리는 추후 추가·삭제가 쉽게 가능해야 한다.

## 7.2 하단 탭 구조

MVP 하단 탭은 다음 4개 카테고리로 구성한다.

```
답변하기 / 나의 고민 / 채팅 / 순위
```

앱 첫 화면의 기본 탭은 **답변하기**다.
사용자가 특정 탭에 있는 경우 해당 탭 버튼이 하이라이트된다.
마이페이지 위치와 진입 방식은 기존 구조를 유지한다.

하단 바는 고정되어 있으며 화면 스크롤에 따라 움직이지 않는다.
정적 preview에 포함된 모바일 status bar의 시간, 네트워크, 배터리 표시와 화면 최하단 home indicator는 production UI에서 구현하지 않는다.

### 답변하기

- 앱 첫 화면
- 내가 답변해야 하는 고민 목록
- 예제 고민 포함 가능
- 읽은 고민과 읽지 않은 고민은 별도 보관함으로 분리하지 않음
- 답변 완료한 고민은 답변하기 탭에서 즉시 사라짐
- 패스한 고민은 답변하기 탭에서 즉시 사라짐
- 화면 우상단에는 마이페이지 버튼이 들어가며, 누르면 마이페이지 화면으로 이동
- 답변하기 탭의 empty 상태 문구는 `지금은 도착한 고민이 없어요.`로 표시
- 답변하기 탭의 로딩 상태는 spinner로 표시

### 나의 고민

- 내가 작성한 고민 목록
- 내가 받은 답변 확인
- 읽은 답변과 읽지 않은 답변은 별도 보관함으로 분리하지 않음
- 답변 좋아요/싫어요/코멘트 입력
- 화면 우상단에는 마이페이지 버튼이 들어가며, 누르면 마이페이지 화면으로 이동
- 고민 작성 진입점은 나의 고민 탭의 우측 하단 메시지 버튼이다.
- 나의 고민 탭의 empty 상태 문구는 `첫 고민을 남겨보세요.`로 표시
- 나의 고민 탭의 로딩 상태는 spinner로 표시

나의 고민 목록에서는 사용자가 작성한 고민의 첫 번째 유효 카테고리, 작성 시각, LLM 한 줄 요약, 현재 게시자에게 보이는 답변 수를 표시한다.
답변 수는 `n명이 답변했어요`로 표시한다.
답변이 0개인 경우에는 `아직 답변이 없어요.`로 표시한다.

답변 수에 포함하는 항목:

- 현재 게시자에게 보이는 사람 답변
- 현재 게시자에게 보이는 AI 답변

답변 수에서 제외하는 항목:

- 싫어요로 숨겨진 답변
- 운영자가 숨김 처리한 답변
- 기타 사용자-facing read model에서 제외된 답변

### 채팅

- 좋아요 코멘트 이후 생성된 1대1 채팅 목록
- 답변 단위 채팅방 진입
- 새 채팅방 및 새 메시지 확인
- 채팅방 내 메시지 송수신
- 채팅방 내 차단 및 신고

### 순위

- 누적 순위 확인
- 이 달의 순위 확인
- 1등부터 10등까지 표시
- 각 순위 항목의 닉네임과 받은 하트 수 표시

### 마이페이지

- 프로필 summary
    - 닉네임
    - 받은 좋아요/하트 총합
- 받은 좋아요/하트 총합의 내부 필드는 `helpedCount`를 사용한다.
- 관심 분야 수정
- 내가 쓴 답변 목록 preview
- 좋아요/코멘트 받은 답변 표시
- 푸시 알림 설정
- 개인정보처리방침
- 로그아웃
- 탈퇴

성별은 MVP에서 수정하지 않는다.
성별 수정 UI는 MVP 제외이며, PRD의 마이페이지 수정 기능은 **관심 분야 수정**으로 한정한다.
나이는 내부 저장 및 운영 분석용 데이터로 유지하지만, MVP 마이페이지에는 표시하지 않는다.

마이페이지의 내가 쓴 답변 preview에는 내가 다른 사용자의 고민에 남긴 답변이 표시된다.
각 항목에는 고민 카테고리와 답변 생성 시각을 표시한다.
`전체보기`를 누르면 내가 쓴 답변 목록 화면으로 이동한다.

내가 쓴 답변 목록에서는 내가 작성한 모든 답변을 동일한 형식으로 표시한다.
좋아요를 받은 답변은 답변 박스 우측 상단에 하트를 표시한다.
좋아요만 있고 코멘트가 없으면 하트만 표시한다.
좋아요 코멘트가 있으면 내가 남긴 답변 아래에 더 작은 폰트로 코멘트를 1개 표시한다.
싫어요를 받은 답변은 답변자에게 피드백 없음처럼 표시되며, 싫어요 여부와 싫어요 코멘트는 노출하지 않는다.
예제 고민 답변에 자동 좋아요가 붙은 경우에도 내가 쓴 답변 목록에 하트를 표시한다.

관심 분야 수정 화면에서는 온보딩과 동일한 관심 분야 목록을 3열로 표시한다.
관심 분야 버튼은 글자 수에 따라 크기가 변하지 않는다.
관심 분야를 0개 선택한 상태에서 저장을 누르면 `1개 이상의 관심 분야를 선택해주세요.`를 표시한다.
저장에 성공하면 마이페이지로 이동한다.
저장에 실패하면 기존 선택값을 유지한다.
관심 분야 수정은 향후 매칭 후보 선정에 즉시 반영되지만, 이미 생성된 delivery에는 영향을 주지 않는다.

알림 설정은 토글 버튼을 통해 제어한다.
토글을 켜면 브라우저 Notification 권한 요청과 FCM/Web Push registration을 시도한다.
토글을 끄면 앱의 푸시 수신 동의 및 토큰 등록을 해제한다.
브라우저 또는 OS 수준의 권한 자체는 브라우저 정책상 앱이 직접 회수하지 못할 수 있으므로, 필요한 경우 사용자가 브라우저 설정에서 직접 변경해야 한다.

### 마이페이지 더보기 범위

접근 가능한 정책 문서는 개인정보처리방침 하나로 한정한다.
운영정책, 이용 안내 전용 화면, 이용약관, 앱처럼 사용하기 안내는 MVP에서 제외한다.

마이페이지에서 접근 가능한 항목:

- 관심 분야 수정
- 개인정보처리방침
- 로그아웃
- 탈퇴

마이페이지에서 제외하는 항목:

- 운영정책
- 이용 안내 전용 화면
- 이용약관
- 앱처럼 사용하기
- 로그인 화면의 정책/약관 링크

## 7.3 첫 화면: 답변하기

앱 첫 화면은 **답변하기** 탭이다.

답변하기 탭에는 사용자가 답변해야 할 고민 목록이 표시된다.

이 목록은 푸시 알림과 무관하게 항상 서버 상태를 기준으로 갱신된다.
사용자가 알림 권한을 거부했거나 푸시 토큰이 없어도, 앱에 접속하면 자신에게 전달된 고민을 확인할 수 있다.

고민 게시자는 답변자가 자신의 고민을 읽었는지 절대 알 수 없다.

답변자가 답변을 제출하면 해당 고민은 답변하기 탭에서 즉시 사라진다.
답변 완료 항목은 답변 탭 active 고민 10개 제한에도 포함하지 않는다.

각 고민 박스에는 다음 정보를 표시한다.

- 첫 번째 유효 카테고리
- 생성 시각
- LLM 한 줄 요약
- 건너뛰기 버튼

카테고리 박스에는 `validCategories`의 0번 인덱스 값을 표시한다.
`validCategories`가 비어 있고 매칭 fallback 카테고리가 `잡담`인 경우 화면에도 `잡담`을 표시한다.
무효 카테고리는 사용자-facing 화면에 표시하지 않는다.
카테고리 칩은 관심 분야 중 가장 긴 표시 텍스트 길이에 맞춘 기본 고정 가로 길이를 사용하며, 카테고리 글자 수에 따라 칩 크기가 달라지지 않게 한다.

생성 시각은 `createdAt` 필드를 사용자 로컬 타임존 기준으로 계산해 표시한다.
표시 규칙은 다음과 같다.

- 1분 미만: `방금 전`
- 1시간 미만: `n분 전`
- 날짜가 바뀌기 전: `n시간 전`
- 사용자 로컬 타임존 기준 날짜가 바뀐 경우: `YYYY.MM.DD`

고민 박스를 누르면 답변 작성 화면으로 이동한다.
고민 박스 우측 상단의 건너뛰기 버튼은 패스 기능과 동일한 도메인 동작이다.
건너뛰기 버튼을 누르면 확인 모달 없이 즉시 답변하기 탭에서 해당 고민이 사라진다.
건너뛰기 사실은 고민 게시자에게 알려지지 않는다.

답변하기 탭에 표시할 고민이 없는 경우 `지금은 도착한 고민이 없어요.`를 표시한다.
로딩 상태는 spinner로 표시한다.
네트워크 오류 상태는 PRD의 제품 정책이 아니라 구현 세부로 관리한다.

---

## 7.4 고민 작성

### 진입 위치

고민 작성은 **나의 고민** 탭의 우측 하단 메시지 버튼에서 시작한다.

앱의 기본 진입 탭은 계속 답변하기다.

### 입력 방식

고민글은 제목 없이 본문만 작성한다.

고민 작성 화면의 상단 좌측 뒤로 가기 버튼을 누르면 나의 고민 탭으로 이동한다.
`고민 작성` 텍스트는 버튼이 아니다.

고민 작성 텍스트 박스에는 기본 상태에서 연필 모양 그래픽과 `당신의 솔직한 이야기를 들려주세요` 문구를 표시한다.
사용자가 텍스트를 입력하기 시작하면 연필 모양 그래픽과 안내 문구는 보이지 않는다.

글자 수 제한:

- 최소 글자 수 제한 없음
- 단, 빈 문자열 또는 공백만 있는 입력은 제출 불가
- 최대 1000자

고민 게시자는 카테고리를 직접 선택하지 않는다.
LLM이 고민 내용을 분석해 카테고리를 추론한다.

LLM이 반환한 카테고리는 모두 DB에 저장한다.
유효 카테고리는 매칭에 사용하고, 무효 카테고리는 내부 평가용으로 보존한다.

LLM이 유효 카테고리 2개와 무효 카테고리 1개를 반환한 경우:

- 세 카테고리 모두 DB에 저장
- 매칭에는 유효 카테고리 2개만 사용

LLM이 무효 카테고리만 반환한 경우:

- 내부 평가용으로 무효 카테고리는 저장
- 매칭에는 `잡담`으로 fallback
- 사용자-facing 화면에도 `잡담`을 표시

모든 사용자-facing 고민 카드와 답변 작성 화면에는 LLM 한 줄 요약이 표시되어야 한다.
LLM 한 줄 요약은 필수 생성 대상이다.
요약 생성에 실패한 경우 사용자 고민 원문 앞 20자를 슬라이스한 뒤 `...`를 붙인 값을 fallback 요약으로 사용한다.
예시는 다음과 같다.

```
01234567890123456789...
```

### 게시 직후 사용자 피드백

사용자가 고민을 제출하고 필터링을 통과하면 토스트 대신 성공 확인 화면을 표시한다.
성공 확인 화면에서는 확인 버튼을 제외한 상호작용 가능한 버튼을 제공하지 않는다.

확인 버튼을 누르면 사용자는 나의 고민 탭으로 이동한다.
작성한 고민 상세 화면으로 자동 이동하지 않는다.

제출 성공 직후 매칭 또는 delivery 생성이 아직 진행 중이고 답변이 없는 경우, 나의 고민 목록에서 해당 고민은 `아직 답변이 없어요.` 상태로 표시한다.

필터링 실패 시에는 성공 확인 화면으로 이동하지 않는다.
사용자는 고민 작성 화면에 그대로 남고, 작성 중이던 글은 삭제되지 않는다.
화면은 작성 화면의 틀을 유지하되, 사유별 안내 문구를 표시한다.

이번 PRD에서는 기존 3분 유예 타이머를 폐기한다.
고민 게시 후 수정/삭제도 제공하지 않는다.

고민은 제출 즉시 서버 처리 대상으로 넘어가며, 매칭과 delivery 생성이 진행된다.

---

## 7.5 고민 수정/삭제 정책

사용자가 작성한 고민은 게시 직후부터 수정할 수 없다.
삭제도 할 수 없다.

단, 운영자는 DB에서 수동으로 특정 고민을 숨김 처리할 수 있다.
운영자 수동 처리 정책은 MVP 범위에 포함하되, 관리자 페이지는 MVP에서 제외한다.

운영자가 숨김 처리한 고민은 게시자와 답변자 모두의 사용자-facing read model에서 사라진다.
숨김 처리된 고민은 답변하기 탭, 나의 고민, 고민 상세, 내가 쓴 답변 목록의 원문 참조에서 노출하지 않는다.

---

## 7.6 답변 작성

답변자는 답변하기 탭에서 자신에게 전달된 고민을 선택해 답변할 수 있다.

답변 작성 화면 상단 좌측 뒤로 가기 버튼을 누르면 답변하기 탭으로 이동한다.
이때 작성 중이던 답변 draft는 폐기한다.

답변 작성 화면에는 다음을 표시한다.

- LLM이 요약한 한 줄 고민
- 첫 번째 유효 카테고리
- 고민 생성 시각
- 답변 입력 텍스트 박스

고민 요약 박스를 누르면 원문 확인 overlay가 답변 작성 화면 위에 뜬다.
원문 확인 overlay에는 첫 번째 유효 카테고리, LLM 한 줄 요약, 고민 게시자의 원문을 표시한다.
`닫기`를 누르면 답변 작성 화면으로 돌아간다.
원문 확인 overlay를 열었다 닫아도 사용자가 작성 중이던 답변 draft는 유지되어야 한다.

답변 입력 텍스트 박스에는 기본 상태에서 연필 아이콘과 `고민자에게 따뜻한 말을 전달해주세요!` 문구를 표시한다.
아이콘과 안내 문구는 겹쳐서는 안 된다.
사용자가 답변 작성을 시작하면 연필 아이콘과 안내 문구는 보이지 않는다.

답변은 한 고민당 한 번만 작성할 수 있다.

답변자는 고민 게시자의 성별, 나이, 관심 분야, 프로필 정보를 볼 수 없다.
게시자와 답변자는 서로 완전 익명으로 처리된다.

답변 제출 시 AI 필터링을 통과해야 한다.
부적절한 답변은 전송되지 않는다.

글자 수 제한:

- 최소 글자 수 제한 없음
- 단, 빈 문자열 또는 공백만 있는 입력은 제출 불가
- 최대 1000자

답변 제출에 성공하면 사용자는 성공 확인 화면으로 이동한다.
성공 확인 화면에서 확인 버튼을 누르면 답변하기 탭으로 이동한다.
내가 쓴 답변 상세 화면은 MVP에서 제공하지 않는다.
답변이 저장된 고민은 답변하기 탭에서 즉시 사라진다.

---

## 7.7 답변 수정/삭제 정책

답변은 제출 직후부터 수정할 수 없다.
삭제도 할 수 없다.

단, 운영자는 DB에서 수동으로 특정 답변을 숨김 처리할 수 있다.
운영자 수동 처리 정책은 MVP 범위에 포함하되, 관리자 페이지는 MVP에서 제외한다.

운영자가 숨김 처리한 답변은 게시자와 답변자 모두의 사용자-facing read model에서 사라진다.
숨김 처리된 답변은 답변 확인 화면, 받은 답변 목록, 내가 쓴 답변 목록에서 노출하지 않는다.

---

## 7.8 읽음 처리

읽음 처리는 사용자-facing 메시지가 아니다.

즉, 사용자에게 다음과 같은 문구는 표시하지 않는다.

```
읽음
상대방이 읽었습니다
상대방이 아직 읽지 않았습니다
```

읽음 여부는 내부 DB에서만 관리한다.

사용자는 다음을 절대 알 수 없다.

- 내 고민을 답변자가 읽었는지 여부
- 내가 작성한 답변을 고민 게시자가 읽었는지 여부

### 고민 읽음

답변자가 답변 작성 화면에 진입하는 순간 해당 고민은 내부적으로 read 처리된다.

체류 시간 조건은 없다.
read 처리는 되돌릴 수 없다.

### 답변 읽음

게시자가 답변 확인 화면에 진입하면, 그 시점에 존재하는 답변들이 내부적으로 read 처리된다.

이후 새로 도착한 답변은 unread 상태로 유지된다.
게시자가 다시 답변 확인 화면에 들어가야 새 답변이 read 처리된다.

---

## 7.9 좋아요, 싫어요, 후기 코멘트

게시자는 자신이 받은 답변에 대해 좋아요 또는 싫어요를 누를 수 있다.

하나의 답변에는 좋아요와 싫어요 중 하나만 선택할 수 있다.
선택 후 취소하거나 변경할 수 없다.

게시자가 자신의 고민 상세 화면에 들어가면, 상단에는 자신의 고민이 표시되고 그 아래에 답변 박스들이 표시된다.
답변이 0개인 경우에는 별도 empty 문구를 표시하지 않고, 내 고민만 보이며 그 아래에는 답변 박스 없이 배경만 보인다.

좋아요/싫어요 코멘트 입력 UI는 구현자가 기존 제품 UI 톤에 맞춰 작성한다.

### 좋아요

좋아요를 누르면 해당 답변자의 내부 `helpedCount`가 1 증가한다.
좋아요는 누르는 즉시 확정된다.
사용자 화면에서는 `helpedCount`를 받은 좋아요/하트 총합으로 표시한다.

좋아요를 누른 경우 선택적으로 코멘트를 남길 수 있다.
좋아요 클릭 후 코멘트 입력 창을 띄울 수 있으며, 사용자는 코멘트를 제출하거나 건너뛸 수 있다.
코멘트를 건너뛰어도 좋아요 자체는 취소되지 않는다.
좋아요 코멘트는 답변자 본인에게만 공개된다.

좋아요 코멘트 글자 수 제한:

- 최소 글자 수 제한 없음
- 단, 빈 문자열 또는 공백만 있는 입력은 제출 불가
- 최대 1000자

좋아요를 누른 뒤 즉시 코멘트를 남기지 않아도 된다.
좋아요를 누른 답변은 숨겨지지 않으므로, 사용자는 나중에 앱을 다시 켜서 해당 답변에 코멘트를 남길 수 있다.
좋아요 코멘트는 답변 하나당 1개만 허용한다.
좋아요와 좋아요 코멘트가 모두 있는 답변은 답변자가 채팅을 시작할 수 있는 대상이 된다.

### 싫어요

싫어요는 내부 평가 전용 부정 피드백이다.

- 답변자에게 절대 보이지 않는다.
- 답변자는 게시자가 아직 피드백을 누르지 않은 것으로 본다.
- 싫어요를 누른 답변은 게시자 화면에서 완전히 숨겨진다.
- 싫어요 코멘트는 선택이다.
- 싫어요 코멘트는 운영자만 볼 수 있다.
- 싫어요에 따른 답변자 불이익은 MVP에서 구현하지 않는다.

싫어요를 누르면 코멘트 입력 창을 띄울 수 있다.
싫어요 자체는 누르는 즉시 취소 불가능한 상태로 확정된다.
다만 답변 박스는 코멘트 입력 창이 종료된 뒤 게시자 화면에서 숨겨진다.
코멘트 입력 창에서 취소하거나 건너뛰어도 싫어요 자체는 취소되지 않는다.

싫어요 코멘트 글자 수 제한:

- 최소 글자 수 제한 없음
- 단, 빈 문자열 또는 공백만 있는 입력은 제출 불가
- 최대 1000자

싫어요에 코멘트를 남기지 않고 코멘트 입력 창을 종료하면, 해당 답변은 이미 숨겨지기 때문에 이후 코멘트를 추가로 남길 수 없다.
이는 의도된 정책이다.
싫어요 코멘트는 답변 하나당 1개만 허용한다.

좋아요/싫어요 코멘트는 모두 AI 필터링을 거친다.

---

## 7.10 helpedCount

답변자가 받은 좋아요는 내부 `helpedCount`에 반영된다.
사용자 화면에서는 `helpedCount`를 받은 좋아요/하트 총합으로 표시한다.

- helpedCount는 줄어들지 않는다.
- 탈퇴자가 남긴 좋아요도 유지된다.
- AI 답변에 대한 좋아요는 helpedCount에 반영하지 않는다.
- 예제 고민에 대한 답변은 helpedCount에 반영한다.

예제 고민은 튜토리얼 역할을 하므로, 사용자가 예제 고민에 답변하면 반드시 좋아요가 돌아온다.
이를 통해 초기 사용자도 답변 경험과 성취감을 얻을 수 있다.

예제 고민 좋아요는 답변 제출 후 5~15분 사이 랜덤 지연 후 생성된다.
예제 고민 좋아요에는 코멘트를 자동 생성하지 않는다.
예제 고민 답변에 자동 좋아요가 붙은 경우에도 내가 쓴 답변 목록에는 하트가 표시된다.

## 7.11 순위

순위 탭에서는 사용자들이 받은 좋아요/하트 개수를 기준으로 1등부터 10등까지 표시한다.

순위는 다음 두 종류로 나뉜다.

- 누적 순위
- 이 달의 순위

누적 순위는 활성 사용자의 `helpedCount` 기준으로 계산한다.
이 달의 순위는 KST 기준 해당 월에 발생한 좋아요 이벤트 수를 기준으로 계산한다.

탈퇴자와 비활성 사용자는 순위에서 제외한다.
탈퇴자가 남긴 좋아요는 기존 정책대로 받은 사용자의 누적 `helpedCount`에는 유지되지만, 탈퇴자 본인은 순위 표시 대상이 아니다.

같은 좋아요 수를 가진 사용자는 공동 순위로 표시한다.
공동 순위가 발생해도 순위 탭에는 최대 10명까지만 표시한다.

각 순위 항목에는 다음 정보를 표시한다.

- 순위
- 닉네임
- 받은 하트 수

순위 탭은 닉네임 공개의 예외 영역이다.
고민, 답변, 채팅 화면에서는 기존 익명 정책을 유지한다.

## 7.12 채팅, 차단 및 신고

채팅은 고민 게시자와 고민 답변자가 좋아요 코멘트를 통해 서로 잘 맞는다고 판단한 뒤 후속 대화를 이어갈 수 있는 기능이다.

채팅방은 답변 단위로 생성한다.
같은 두 사용자가 여러 답변에서 연결되더라도, 각 답변의 후속 대화는 별도 채팅방으로 관리한다.

채팅 시작 조건은 다음 순서가 모두 충족된 경우다.

1. 고민 게시자가 답변에 좋아요를 누른다.
2. 고민 게시자가 좋아요 코멘트를 남긴다.
3. 고민 답변자가 해당 코멘트에 응답하고 싶을 때 채팅을 시작한다.

채팅 시작 권한은 고민 답변자에게만 있다.
고민 게시자는 먼저 채팅을 시작할 수 없다.
코멘트 없는 좋아요만으로는 채팅을 시작할 수 없다.

채팅 화면에서는 상대방을 익명으로 표시한다.
상대방의 닉네임, 성별, 나이, 관심 분야, 프로필 정보는 채팅 화면에 노출하지 않는다.

채팅 메시지는 AI 필터링을 통과해야 전송된다.
부적절한 채팅 메시지는 전송되지 않으며, 사용자는 사유별 안내 문구를 받는다.

채팅 화면에는 상대방에 대한 `차단 및 신고` 버튼을 둔다.
사용자가 이 버튼을 누르면 신고 사유 카테고리와 상세 내용을 입력할 수 있다.

신고 사유 카테고리는 다음 5개로 시작한다.

1. 욕설/괴롭힘
2. 성적 표현
3. 개인정보 요구
4. 스팸/홍보
5. 기타

신고가 제출되면 해당 채팅방은 신고자 화면에서 삭제된다.
상대방 화면에서는 즉시 삭제하지 않는다.
신고 이후 신고자는 해당 채팅방에서 더 이상 메시지를 주고받지 않는다.

신고 사유, 상세 내용, 해당 채팅방의 전체 대화는 운영자 확인을 위해 보존한다.
운영자는 신고된 채팅의 전체 대화와 신고 내용을 열람할 수 있다.

## 8. 매칭 정책

## 8.1 기본 매칭

고민이 제출되고 필터링을 통과하면 서버가 답변자를 매칭한다.

기본 목표는 하나의 고민에 대해 **5개의 delivery**를 생성하는 것이다.

5명 중 4명은 기존 알고리즘에 따라 매칭한다.
5명 중 1명은 관심 분야, 좋아요 수, 성별 조건을 사용하지 않는 랜덤 매칭으로 선택한다.
서비스 초기 상황, 유저가 5명 미만인 경우 모든 유저들에게 전송한다.

단, 랜덤 매칭도 다음 기본 제한은 지켜야 한다.

- 고민 작성자 본인 제외
- 이미 같은 고민을 받은 사용자 제외
- 탈퇴/비활성 사용자 제외
- 답변 탭 active 고민 10개 미만 사용자만 포함

푸시 토큰이 없거나 알림 권한을 거부한 사용자도 매칭 대상에 포함된다.

---

## 8.2 답변 탭 10개 제한

기존 하루 3개 수신 제한 정책은 폐기한다.

대신 한 사용자의 **답변하기 탭에 답변을 기다리는 고민이 10개를 초과하지 않도록 제한**한다.

10개 제한에 포함되는 항목:

- 아직 답변하지 않은 active delivery
- 읽었지만 아직 답변하지 않은 active delivery
- 패스하지 않은 delivery

10개 제한에 포함하지 않는 항목:

- 이미 답변한 고민
- 패스한 고민
- 예제 고민
- 운영자가 숨김 처리한 고민

사용자가 답변하기 탭에 active 고민 10개를 가지고 있으면 신규 매칭 대상에서 제외된다.
사용자가 답변하거나 패스하면 active 고민 수가 줄어들고 다시 매칭 대상이 될 수 있다.

이미 답변한 고민은 답변하기 탭에서 즉시 사라진다.
따라서 답변 완료 상태를 답변하기 탭에 계속 표시하지 않는다.

---

## 8.3 패스 기능

답변하기 싫은 고민은 패스할 수 있다.

사용자는 답변하기 탭의 고민 박스 우측 상단에 있는 **건너뛰기** 버튼을 눌러 해당 고민을 패스한다.
스와이프 기반 패스 UI는 MVP 요구사항으로 사용하지 않는다.

건너뛰기와 패스는 같은 도메인 동작이다.

패스 정책:

- 건너뛰기 버튼을 누르면 확인 모달 없이 즉시 답변하기 탭에서 사라진다.
- 패스한 사용자는 같은 고민을 다시 받을 수 없다.
- 패스 사실은 고민 게시자에게 알려지지 않는다.
- 패스는 전체 최대 15개 delivery 슬롯에 포함된다.
- 패스된 고민은 다른 사용자에게 재매칭된다.

패스 기능은 사용자가 답변하기 싫은 고민을 직접 정리할 수 있게 하며, 답변 탭 10개 제한의 부작용을 줄이기 위한 장치다.

---

## 8.4 카테고리 기반 매칭

LLM은 고민 내용을 읽고 1개 이상의 카테고리를 추론한다.

이후 각 답변자 후보의 관심 분야와 LLM 추론 카테고리의 교집합을 계산한다.

교집합이 많은 그룹부터 후보를 선택한다.

동률 또는 같은 그룹 내에서는 다음 기준을 적용한다.

1. 좋아요 수
2. 같은 성별 우선
3. 랜덤 추출

매칭 우선순위 전체는 다음과 같다.

```
1순위: 관심 분야
2순위: 좋아요 수
3순위: 성별
```

나이는 MVP 매칭 우선순위에 포함하지 않는다.

---

## 8.5 랜덤 매칭 규칙

초기 매칭에서는 5명 중 1명을 랜덤 매칭으로 포함한다.

재매칭에서도 “5명 중 1명 랜덤” 규칙을 유지한다.
다만 재매칭 시에는 기존 답변 여부와 기존 랜덤 매칭 포함 여부에 따라 다음과 같이 처리한다.

예를 들어 초기 5명 중 n명이 답변한 상태라고 하자.

- n명 안에 랜덤 매칭 사용자가 포함된 경우
    - 이미 랜덤 슬롯이 답변으로 소진된 것으로 본다.
    - 남은 5-n명은 랜덤성 없이 기존 알고리즘으로 재매칭한다.
- n명 안에 랜덤 매칭 사용자가 포함되지 않은 경우
    - 남은 5-n명 중 1명은 랜덤 매칭으로 포함한다.
    - 나머지는 기존 알고리즘으로 재매칭한다.

즉, 각 5명 묶음 기준으로 랜덤 매칭 1명이라는 구조를 유지하되, 이미 답변한 랜덤 사용자가 있는 경우에는 추가 랜덤 슬롯을 만들지 않는다.

---

## 8.6 재매칭

재매칭 시간은 **8시간**으로 줄인다.

기본 흐름:

```
초기 5명 전달
→ 8시간 경과
→ 답변하지 않은 슬롯을 다른 사용자에게 재매칭
→ 다시 8시간 경과
→ 여전히 답변하지 않은 슬롯을 다른 사용자에게 재매칭
→ 다시 8시간 경과
→ 총 24시간 경과
→ 총 15개 인간 delivery 슬롯 중 아무도 답변하지 않은 경우 AI 답변 생성
```

최대 delivery 수는 누적 기준 15개다.

```
초기 delivery 5개
1차 재매칭 최대 5개
2차 재매칭 최대 5개
총 최대 15개
```

기존 전달자가 8시간 이후 늦게 답변하는 것도 허용한다.
따라서 하나의 고민은 5개를 초과하는 사람 답변을 받을 수 있다.

---

## 8.7 AI 답변 생성 조건

AI 답변은 사람이 충분히 답변하지 못한 상황을 보완하기 위한 fallback이다.

AI 답변은 다음 조건에서만 생성된다.

```
초기 5명 + 1차 재매칭 최대 5명 + 2차 재매칭 최대 5명,
총 최대 15개 인간 delivery 슬롯 중
24시간 동안 단 한 건의 사람 답변도 서버에 저장되지 않은 경우
AI 답변 생성.
```

여기서 “답변 도착”은 사람이 답변을 작성해 서버에 저장한 시점을 기준으로 한다.

싫어요로 숨겨진 답변도 답변 도착으로 간주한다.
따라서 싫어요로 숨겨진 답변이 하나라도 있으면 AI 답변 생성 조건에는 해당하지 않는다.

AI 답변은 한 고민당 2개 이상 전달되지 않는다.
AI 답변이 이미 전달된 경우에는 24시간 경과 후 추가 AI 답변을 생성하지 않는다.

AI 답변 도착 시에도 게시자에게 일반 답변과 동일하게 “내 고민에 답변이 도착했어요” 푸시를 보낸다.

AI 답변은 사용자에게 사람 답변과 구분되지 않는다.
DB에는 AI 답변 여부를 저장한다.

AI 답변은 전문 상담처럼 보이지 않아야 하며, 일반 사용자의 짧은 공감 답변처럼 작성되어야 한다.

AI 답변도 사용자에게 표시되기 전 AI 필터링을 다시 거친다.

---

## 9. 예제 고민

초기 사용자 부족 문제를 해결하기 위해 예제 고민 데이터를 제공한다.

예제 고민은 사람이 미리 작성한다.
각 관심 분야별로 예제 고민을 준비한다.

온보딩 직후 사용자가 선택한 관심 분야에 맞춰 예제 고민이 표시된다.

- 최대 5개까지 표시
- 사용자가 선택한 관심 분야가 5개보다 적으면 그 수만큼만 표시
- 관심 분야를 수정해도 예제 고민은 추가 생성되지 않음
- 예제 고민은 온보딩 직후 한 번만 나타남
- 이후 새로운 예제 고민은 나타나지 않음

예제 고민은 실제 고민과 동일하게 보인다.
사용자는 예제 고민과 실제 고민의 차이를 알 수 없어야 한다.
예제 고민임을 나타내는 라벨, 튜토리얼 배지, 샘플 표시, 관리자 전용 문구를 사용자-facing UI에 노출하지 않는다.

예제 고민에는 게시자 attribute가 없거나, 예제 고민 전용 placeholder를 사용한다.

예제 고민에 대한 답변도 실제 답변과 동일하게 AI 필터링을 거친다.
운영자는 예제 고민 답변을 품질 평가용으로 열람할 수 있다.

예제 고민에 대한 답변은 답변자의 “내가 쓴 답변 목록”에 남는다.
예제 고민 답변은 실제 답변과 동일하게 표시한다.
사용자-facing UI에서는 예제 고민 답변인지 구분하지 않는다.

예제 고민 답변에는 반드시 좋아요가 돌아온다.
좋아요는 답변 제출 후 5~15분 사이 랜덤 지연 후 생성된다.
예제 고민 좋아요에는 자동 코멘트를 생성하지 않는다.

예제 고민 답변에 대한 좋아요는 helpedCount에 반영된다.

예제 고민은 튜토리얼로 동작한다.
사용자는 예제 고민과 실제 고민의 차이를 느끼지 못해야 하며, 예제 고민 답변을 통해 답변 경험과 성취감을 얻을 수 있어야 한다.

## 10. 알림 정책

Qling은 현재 코드 기준의 PWA/Web Push 구조를 유지한다.

알림 종류는 다음 다섯 가지다.

1. 새 고민 도착
2. 내 고민에 답변 도착
3. 내 답변에 좋아요
4. 새 채팅방 시작
5. 새 채팅 메시지 도착

피드백 코멘트만으로는 알림을 보내지 않는다.
싫어요 알림은 보내지 않는다.
수정/삭제 기능이 없으므로 사용자 수정/삭제 알림도 없다.

마이페이지에는 푸시 알림 설정 기능을 둔다.
푸시 알림 설정은 단순 안내 문구가 아니라, 브라우저 Notification 권한 요청과 FCM/Web Push registration을 시도하는 실제 기능이다.

알림 설정은 토글 버튼으로 제공한다.
토글을 켜면 알림 권한 요청과 푸시 registration을 시도한다.
토글을 끄면 앱의 푸시 수신 동의와 등록된 푸시 토큰을 해제한다.
브라우저 또는 OS 권한 자체를 앱이 직접 회수할 수 없는 경우에는 사용자가 브라우저 설정에서 직접 변경해야 한다.

푸시 알림 설정 화면 또는 영역은 다음 상태를 표시한다.

- 권한 허용됨
- 권한 차단됨
- 설정 필요
- 브라우저 설정에서 직접 허용해야 하는 경우

알림 권한을 거부한 사용자도 매칭 대상에 포함한다.
푸시 토큰이 없는 사용자도 매칭 대상에 포함한다.

delivery 생성은 성공했지만 푸시 발송만 실패한 경우, 고민 전송은 성공으로 처리한다.

invalid push token은 자동 삭제한다.

## 11. 필터링 정책

Qling은 다음 대상에 AI 필터링을 적용한다.

- 고민 게시
- 답변 작성
- 좋아요 코멘트
- 싫어요 코멘트
- 채팅 메시지
- 예제 고민에 대한 답변
- AI 답변 생성 결과

기존 수정 기능이 폐기되었으므로, 고민 수정본 또는 답변 수정본에 대한 필터링은 필요하지 않다.

필터링은 사용자가 제출하는 즉시 동기 처리한다.
필터링을 통과한 경우에만 다음 단계로 진행한다.

고민 게시의 경우:

```
필터링 통과 → 매칭 및 delivery 생성 진행
```

필터링 실패 시 사용자는 사유별 안내 문구를 받는다.
작성 중이던 글은 삭제되지 않는다.

고위험 고민은 게시를 차단하고, 일반 부적절 문구와 함께 도움 안내 문구를 제공한다.

필터링 유형은 6가지로 나뉜다.

```
공격적, 혐오, 욕설 표현
성적 표현
개인정보 포함
스팸, 홍보성 표현
자해, 자살 위험 표현 - 관련 기관 소개 문구
범죄, 폭력 피해 표현 - 관련 기관 소개 문구
```

필터링 로그와 차단 원문은 운영 및 품질 평가를 위해 영구 보존한다.

## 12. 익명성과 데이터 정책

Qling은 외부 사용자에게 다음 가치를 강조한다.

```
이름, 연락처, 프로필 사진 없이 익명으로 고민을 나눌 수 있다.
```

다만 내부적으로는 서비스 운영을 위해 다음 데이터를 수집·저장한다.

- 닉네임
- 성별
- 나이
- 관심 분야
- 고민 본문
- 고민 한 줄 요약
- 답변 본문
- 좋아요/싫어요
- 좋아요 발생 시각
- 후기 코멘트
- 채팅 본문
- 채팅 신고 사유와 상세 내용
- 푸시 토큰
- 필터링 로그
- 매칭 결과
- 푸시 발송 결과
- AI 답변 여부
- 예제 고민 여부
- 패스 여부
- 내부 read 상태

닉네임은 본인의 마이페이지 프로필 summary와 순위 탭에서 표시한다.
순위 탭은 닉네임 공개의 예외 영역이다.
나이는 다른 사용자에게 표시하지 않으며, MVP 마이페이지 화면에도 표시하지 않는다.

고민, 답변, 채팅 맥락에서 사용자끼리는 서로의 닉네임, 성별, 나이, 관심 분야, 프로필 정보를 볼 수 없다.
게시자와 답변자는 순위 탭을 제외한 고민/답변/채팅 맥락에서 익명으로 처리된다.

운영자는 원문과 내부 로그를 열람할 수 있다.

개인정보처리방침은 별도 문서로 관리한다.

- `docs/privacy_policy.md`

MVP 구현 단계에서는 위 파일을 빈 파일로 둘 수 있다.
다만 앱의 개인정보처리방침 화면은 위 문서를 source of truth로 삼는 구조로 확장 가능해야 한다.
문서가 비어 있을 경우 사용자 화면에는 `정책을 준비 중입니다.`를 표시한다.

운영정책과 이용약관은 MVP에서 제외한다.

## 13. 탈퇴 정책

사용자가 탈퇴하면 `users/{uid}` 문서는 완전 삭제하지 않고 `deleted` 상태로 유지한다.

요구사항에서 말하는 계정 삭제는 사용자 접근권 삭제 및 계정 비활성화를 의미한다.
DB의 사용자 문서는 기존 정책대로 `deleted` 상태로 보존한다.

탈퇴 후 해당 사용자는 더 이상 매칭 대상이 되지 않는다.
푸시 토큰은 제거되며 알림을 받지 않는다.

탈퇴 후에도 게시 성공한 고민, 답변, 후기, 좋아요/싫어요는 계속 보존한다.

탈퇴자가 작성한 고민 본문은 유지한다.
탈퇴자가 작성한 답변도 게시자에게 계속 보인다.
탈퇴자가 남긴 후기 코멘트와 좋아요도 유지한다.

탈퇴 즉시 해당 사용자는 더 이상 앱에서 활동할 수 없다.
탈퇴 후 콘텐츠 처리는 운영자 수동 조치로만 가능하다.

탈퇴 확인 화면에서 탈퇴 버튼을 누르면 추가 확인 입력 없이 탈퇴를 수행한다.
탈퇴 완료 후 사용자는 로그인 화면으로 이동한다.
탈퇴 확인 화면은 마이페이지 위에 뜨는 overlay/dialog 성격의 화면이며, 배경의 마이페이지와 하단바는 흐릿하게 보일 수 있다.
단, overlay가 열린 상태에서 배경 하단바는 클릭할 수 없다.

## 14. 운영자 권한

운영자는 서비스 운영 및 품질 평가를 위해 내부 데이터를 열람할 수 있다.

운영자가 열람 가능한 정보:

- 닉네임
- 성별
- 나이
- 관심 분야
- 고민 원문
- 답변 원문
- 좋아요/싫어요
- 좋아요 발생 시각
- 좋아요/싫어요 코멘트
- 채팅 본문
- 채팅 신고 사유와 상세 내용
- 필터링 로그
- 차단 원문
- LLM 카테고리 추론 결과
- 유효/무효 카테고리
- 매칭 결과
- 랜덤 매칭 여부
- 패스 여부
- 재매칭 결과
- 푸시 발송 성공/실패 결과
- AI 답변 여부
- 예제 고민 여부
- 내부 read 상태
- 채팅 신고 여부

MVP에서는 관리자 페이지를 구현하지 않는다.
다만 운영자는 DB에서 수동으로 특정 고민/답변을 숨김 처리할 수 있다.
운영자는 신고된 채팅의 전체 대화와 신고 내용을 열람할 수 있다.

운영자 숨김 처리된 고민/답변은 게시자와 답변자 모두의 사용자-facing read model에서 사라진다.

## 15. MVP 제외 범위

MVP에서는 다음 기능을 제외한다.

```
공개 게시판
검색
프로필 공개

좋아요/싫어요 기반 자동 페널티
좋아요/싫어요 취소
좋아요/싫어요 변경
사용자 고민 수정/삭제
사용자 답변 수정/삭제
내가 쓴 답변 상세 화면

이용 안내 전용 화면

프로필 아바타는 MVP 제품 요구사항에 포함하지 않는다.
PRD 요구사항은 프로필 사진 업로드와 프로필 공개가 없다는 점으로 한정한다.

## 16. 주요 사용자 플로우

## 16.1 첫 실행 및 온보딩

1. 사용자가 Qling에 접속한다.
2. 사용자가 닉네임을 입력한다.
3. 서버가 닉네임 중복 여부를 최종 확인한다.
4. 사용자가 성별을 선택한다.
5. 사용자가 나이를 입력한다.
6. 사용자가 관심 분야를 최소 1개 이상 선택한다.
7. 온보딩이 완료된다.
8. 선택한 관심 분야에 맞는 예제 고민이 최대 5개 생성된다.
9. 사용자는 답변하기 탭에서 자신에게 온 고민을 확인한다.

---

## 16.2 고민 게시 플로우

1. 사용자가 나의 고민 탭의 우측 하단 메시지 버튼을 통해 고민 작성 화면으로 이동한다.
2. 최대 1000자의 고민을 작성한다.
3. 빈 문자열 또는 공백만 있는 입력은 제출할 수 없다.
4. 제출한다.
5. AI 필터링을 즉시 수행한다.
6. 필터링 실패 시 작성 화면에 그대로 남고, 사유별 경고를 보여준다.
7. 필터링 통과 시 성공 확인 화면을 보여준다.
8. 사용자가 확인 버튼을 누르면 나의 고민 탭으로 이동한다.
9. 답변이 아직 없는 경우 해당 고민은 `아직 답변이 없어요.` 상태로 표시된다.
10. 서버는 고민 카테고리를 추론한다.
11. 서버는 고민 한 줄 요약을 생성한다.
12. 서버는 답변자 5명을 매칭한다.
13. 답변자에게 delivery가 생성되고 푸시가 발송된다.
14. 게시자는 답변 확인 화면 또는 나의 고민 탭에서 답변 도착을 확인한다.

---

## 16.3 고민 답변 플로우

1. 사용자가 답변하기 탭에 들어간다.
2. 자신에게 전달된 고민 목록을 본다.
3. 고민 박스를 눌러 답변 작성 화면에 진입한다.
4. 사용자가 고민 요약 박스를 누르면 원문 확인 overlay가 뜬다.
5. 사용자가 overlay를 닫으면 답변 작성 화면으로 돌아오며, 작성 중이던 답변 draft는 유지된다.
6. 답변을 작성한다.
7. 빈 문자열 또는 공백만 있는 입력은 제출할 수 없다.
8. AI 필터링을 통과하면 답변이 저장된다.
9. 사용자는 답변 제출 성공 확인 화면으로 이동한다.
10. 확인 버튼을 누르면 답변하기 탭으로 이동한다.
11. 답변한 고민은 답변하기 탭에서 즉시 사라진다.
12. 게시자에게 답변 도착 알림이 간다.
13. 게시자가 답변을 읽고 좋아요 또는 싫어요를 누를 수 있다.
14. 좋아요를 받으면 답변자의 helpedCount가 증가하고, 화면에서는 받은 좋아요/하트 총합이 증가한 것으로 표시된다.

---

## 16.4 패스 플로우

1. 사용자가 답변하기 탭에서 답변하기 싫은 고민을 본다.
2. 해당 고민 박스 우측 상단의 건너뛰기 버튼을 누른다.
3. 확인 모달 없이 해당 고민은 답변하기 탭에서 즉시 사라진다.
4. 해당 사용자는 같은 고민을 다시 받지 않는다.
5. 고민 게시자에게 패스 사실은 알려지지 않는다.
6. 해당 패스는 최대 15개 delivery 슬롯에 포함된다.
7. 서버는 다른 사용자에게 해당 고민을 재매칭한다.

---

## 16.5 피드백 플로우

1. 게시자가 답변 확인 화면에 들어간다.
2. 해당 시점에 존재하는 답변들이 내부적으로 read 처리된다.
3. 게시자는 각 답변에 좋아요 또는 싫어요를 누를 수 있다.
4. 좋아요를 누르면 좋아요가 즉시 확정된다.
5. 좋아요 후 선택적으로 코멘트를 작성하거나 건너뛸 수 있다.
6. 좋아요 코멘트는 나중에 1회까지 추가할 수 있다.
7. 싫어요를 누르면 싫어요가 취소 불가능하게 확정된다.
8. 싫어요 후 선택적으로 내부 평가용 코멘트를 작성하거나 건너뛸 수 있다.
9. 싫어요를 누른 답변은 코멘트 입력 창이 종료된 뒤 게시자 화면에서 숨겨진다.
10. 싫어요 코멘트를 남기지 않고 나가면 이후 코멘트를 추가로 남길 수 없다.
11. 좋아요/싫어요 코멘트는 AI 필터링을 거친다.
12. 좋아요 코멘트는 답변자에게 보인다.
13. 싫어요 코멘트는 답변자에게 보이지 않고 운영자만 볼 수 있다.

---

## 16.6 채팅 플로우

1. 게시자가 답변 확인 화면에서 특정 답변에 좋아요를 누른다.
2. 게시자가 해당 답변에 좋아요 코멘트를 남긴다.
3. 답변자는 내가 쓴 답변 목록 또는 채팅 진입 가능 상태에서 좋아요 코멘트를 확인한다.
4. 답변자가 원하면 해당 코멘트에 응답하는 채팅을 시작한다.
5. 채팅방은 해당 답변 단위로 생성된다.
6. 게시자는 새 채팅방 시작 알림을 받을 수 있다.
7. 양쪽 사용자는 채팅 탭에서 해당 채팅방을 확인한다.
8. 채팅 메시지는 전송 전 AI 필터링을 거친다.
9. 필터링을 통과한 메시지만 상대방에게 전달된다.
10. 새 채팅 메시지가 도착하면 상대방에게 알림을 보낼 수 있다.

---

## 16.7 차단 및 신고 플로우

1. 사용자가 채팅 화면에서 `차단 및 신고` 버튼을 누른다.
2. 사용자가 신고 사유 카테고리를 선택한다.
3. 사용자가 상세 내용을 입력한다.
4. 신고를 제출한다.
5. 신고자 화면에서 해당 채팅방이 삭제된다.
6. 신고자는 해당 채팅방에서 더 이상 메시지를 주고받지 않는다.
7. 신고 사유, 상세 내용, 해당 채팅방의 전체 대화는 운영자 확인을 위해 보존된다.
8. 운영자는 신고된 채팅의 전체 대화와 신고 내용을 열람할 수 있다.

---

## 17. 정책 문서와 마이페이지/더보기 범위

마이페이지/더보기에서 접근 가능한 정책 문서는 다음 하나다.

- 개인정보처리방침

문서의 source of truth는 다음 파일이다.

- `docs/privacy_policy.md`

MVP 구현 단계에서는 위 파일을 빈 파일로 둘 수 있다.
다만 route, navigation, display structure는 문서가 향후 실제 본문을 가질 수 있도록 구성한다.
문서가 비어 있을 경우 사용자 화면에는 `정책을 준비 중입니다.`를 표시한다.

MVP에서는 다음을 별도 route로 만들지 않는다.

- 운영정책 화면
- 이용 안내 전용 화면
- 이용약관 화면
- 앱처럼 사용하기 안내 화면
- 로그인 화면의 정책/약관 링크

---

## 18. 제품 기획 요약

Qling은 공개 게시판형 고민 커뮤니티가 아니라, AI가 고민의 맥락을 파악해 적절한 답변자에게 익명으로 전달하는 비공개 고민 공유 서비스다.

MVP의 핵심은 다음이다.

```
1. 닉네임/성별/나이/관심 분야 기반 온보딩
2. 서버 최종 닉네임 중복 검사
3. 답변하기 중심 첫 화면
4. 답변하기/나의 고민/채팅/순위 4개 하단 탭
5. 고민 작성 진입점은 나의 고민 탭 우측 하단 메시지 버튼
6. 고민 작성 후 즉시 AI 필터링 및 매칭
7. 고민 제출 성공 확인 화면 후 나의 고민 탭 이동
